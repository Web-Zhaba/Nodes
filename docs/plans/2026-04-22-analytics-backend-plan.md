# Следующие шаги: Интеграция реальных данных Аналитики (Фаза 2)

## Цель:
Заменить текущие демо-данные на реальные агрегированные данные пользователя. Устранить визуальные ошибки при рендеринге графиков и их адаптивности. 

---

### Этап 1: Исправление визуальных дефектов графиков (Текущие баги)

**1. Ошибка прозрачности/интенсивности тепловой карты:**
- Текущая проблема: В `ContributionGraph` все 5 уровней закрашены неверными CSS-переменными. Уровень 0 (отсутствие импульсов) использует акцентный `bg-primary`, из-за чего будущие дни полностью закрашены. Несуществующий `bg-brand` не подхватывается дизайн-системой.
- Решение: Обновить массив уровней на `["bg-secondary/40", "bg-primary/30", "bg-primary/50", "bg-primary/80", "bg-primary"]`. Отключить показ ячеек в будущем времени (после `new Date()`), приравняв их уровень к прозрачному.
- Адаптивность: Сейчас `w-2.5 h-2.5` задает фиксированную ширину, которая меньше ширины экрана. Перепишем классы на использование пропорциональной ширины экрана/контейнера, чтобы таблица растягивалась по всей доступной ширине.

**2. Ошибка линий в верхнем графике (Stability):**
- Текущая проблема: Линии не рисуются, хотя точки есть при наведении. Это вызвано тем, что атрибут `stroke` напрямую берет CSS-переменную через `stroke="hsl(var(--primary))"`, этот формат не поддерживается корректно Recharts внутри Shadcn.
- Решение: Использовать стандартную систему Shadcn -> `stroke="var(--color-nodeA)"`. ChartContainer автоматически сгенерирует эти цвета из объекта конфигурации.

### Этап 2: Интеграция с реальными данными (Zustand -> Django)

**1. Подключение Zustand-стора для API-запросов:**
- В файле `useAnalyticsStore.ts` добавить Zustand async action `fetchAnalyticsData(days: number)`:
- Запрос будет обращаться к реализованному в Django `/api/analytics/history`.
- Мы перенесем логику авторизованного Fetch из `useNodesStore` и добавим токен для корректного запроса.

**2. Интеграция с бэкендом Django API:**
- Эндпоинт `get_analytics_history` в `nodes-backend/engine/views.py`:
- На данный момент он отдает заглушку. Необходимо написать сырой SQL запрос в PostgreSQL (аналогично как мы сделали в `process_pulse`) или логику через Django ORM для агрегирования стабильности за каждый день и количества импульсов `request.user`.

### Этап 3: Фильтрация по фокусу (Global vs Node/Core)

**Логика интерактивности:**
- Сбор данных (Fat Payload): Мы запрашиваем весь массив за 30 дней 1 раз.
- Когда `useAnalyticsStore.focusEntity === null`: Графики показывают линии *всех* активных узлов пользователя, а тепловая карта показывает агрегированную сумму всех импульсов за день.
- Когда `useAnalyticsStore.focusEntity = { type: 'node', id: '123' }`: Данные локально фильтруются только для этого узла. Stability-график оставляет только 1 линию, а тепловая карта — только импульсы этого действия. Это достигается фильтрацией данных на стороне React для нулевой задержки (без дополнительных API-запросов).
