## 1. Обзор проекта

### Что такое Nodes?

**Nodes** — персональный трекер привычек с визуализацией в виде сети взаимосвязанных узлов.

### Ключевые концепции

| Концепт | Описание | Пример |
|---------|----------|--------|
| **Узел** | Единица действия (привычка) | "Утренняя зарядка", "Чтение" |
| **Связь** | Взаимовлияние между узлами | "Зарядка → усиливает → Энергия" |
| **Импульс** | Отметка о выполнении | Галочка за сегодня |
| **Стабильность** | Длина стрика (дней подряд) | 15 дней |

### Почему соло-разработка?

```
✅ Полный контроль над продуктом
✅ Нет необходимости согласовывать решения
✅ Можно быстро менять приоритеты
✅ Все знания остаются у вас
✅ Портфолио проект для карьеры
```

## 2. Принципы соло-разработки

### 2.1 Золотые правила

```markdown
1. ❌ НЕ пытайтесь сделать всё сразу
2. ✅ Сделайте одну фичу — запустите — получите фидбек
3. ❌ НЕ оптимизируйте преждевременно
4. ✅ Копируйте готовые решения (не изобретайте велосипед)
5. ❌ НЕ перфекционируйте дизайн
6. ✅ Используйте UI-библиотеки
7. ❌ НЕ пишите тесты для всего (сначала)
8. ✅ Тестируйте только критический путь
9. ❌ НЕ усложняйте архитектуру
10. ✅ Монолит лучше микросервисов для старта
```

### 2.2 Приоритизация (MoSCoW метод)

| Категория | Описание | Доля времени |
|-----------|----------|--------------|
| **Must have** | Без этого продукт не работает | 50% |
| **Should have** | Важно, но можно отложить | 30% |
| **Could have** | Приятно, но не обязательно | 15% |
| **Won't have** | Не делаем в MVP | 5% |

### 2.3 Время разработки

```
Реалистичный график для соло-разработчика:

Будние дни: 1-2 часа/день = 7-10 часов/неделя
Выходные: 4-6 часов/день = 8-12 часов/неделя
─────────────────────────────────────────────────
Итого: 15-22 часа/неделя

При таком темпе:
- MVP: 8-12 недель
- Полная версия: 16-24 недели
```

## 3. Технический стек (бесплатный)

### 3.1 Frontend

```yaml
Фреймворк: React 18+ (бесплатно)
Язык: TypeScript (бесплатно)
Стилизация: Tailwind CSS (бесплатно)
UI компоненты: Shadcn UI (бесплатно, open-source)
Состояние: Zustand (проще Redux, бесплатно)
Роутинг: React Router (бесплатно)
Формы: React Hook Form (бесплатно)
Валидация: Zod (бесплатно)
Графики: Recharts (бесплатно)
Граф/сеть: React Flow (бесплатно)
Сборка: Vite (бесплатно, быстрее CRA)
```

### 3.2 Backend

```yaml
Вариант A (рекомендуется для соло):
  - Supabase (PostgreSQL + Auth + Realtime)
  - Edge Functions (серверная логика)
  - Бесплатный тариф: 500MB БД, 50K запросов/месяц

Вариант B (классический):
  - Django + Django REST Framework
  - PostgreSQL (локально или Railway free tier)
  - Бесплатный хостинг: Railway, Render, Fly.io
```

### 3.3 Инфраструктура

```yaml
Хостинг фронтенда: Vercel (бесплатно)
Хостинг бэкенда: Supabase free tier / Railway free
База данных: Supabase PostgreSQL (бесплатно)
Аутентификация: Supabase Auth (бесплатно)
CI/CD: GitHub Actions (бесплатно)
Домен: можно начать без домена (vercel.app)
Мониторинг: Sentry free tier (5K ошибок/месяц)
Аналитика: Plausible / Umami (self-hosted, бесплатно)
```

### 3.4 Инструменты разработки

```yaml
Редактор: VS Code (бесплатно)
Контроль версий: Git + GitHub (бесплатно)
Дизайн: Figma free tier / Excalidraw
Документация: Notion free tier / GitHub Wiki
Коммуникация: Discord / Telegram (для фидбека)
Тестирование: Playwright (бесплатно)
```

## 8. Риски и решения

### 8.1 Технические риски

| Риск | Вероятность | Влияние | Решение |
|------|-------------|---------|---------|
| Выгорание | 🔴 Высокая | 🔴 Высокое | Делать перерывы, не работать по выходным всегда |
| Потеря мотивации | 🔴 Высокая | 🔴 Высокое | Делить на мелкие задачи, праздновать победы |
| Технические долги | 🟡 Средняя | 🟡 Среднее | Рефакторить по мере роста, не оптимизировать преждевременно |
| База данных переполнится | 🟢 Низкая | 🟡 Среднее | Supabase free = 500MB, хватит на годы |
| Vercel лимиты | 🟢 Низкая | 🟢 Низкое | 100GB bandwidth — очень много для личного проекта |

### 8.2 Решения для выгорания

```markdown
## Правила для соло-разработчика

1. **Максимум 20 часов в неделю**
   - Не больше, иначе выгорите
   
2. **Обязательные выходные**
   - Минимум 1 полный день отдыха в неделю
   
3. **Маленькие победы**
   - Каждый коммит — прогресс
   - Каждая неделя — что-то готово
   
4. **Гибкость**
   - Если не успел на неделе — не страшно
   - План можно менять
   
5. **Публичная ответственность**
   - Вести Twitter/LinkedIn о прогрессе
   - Друзьям рассказывать о целях
```

### 8.3 Когда застрять

```markdown
## Что делать если застрял

1. **Проблема не решается 2+ часа**
   - Сделать перерыв (прогулка, еда, сон)
   - Погуглить ошибку по-другому
   - Спросить в Discord/Reddit/Stack Overflow

2. **Потерял мотивацию**
   - Вспомнить зачем начал
   - Посмотреть на уже сделанное
   - Сделать самую простую задачу для дофамина

3. **Фича слишком сложная**
   - Упростить требование
   - Отложить на пост-MVP
   - Найти готовое решение

4. **Баг не воспроизводится**
   - Добавить больше логов
   - Воспроизвести в инкогнито
   - Спросить у тестировщиков
```

## 9. MVP определение

### 9.1 Что входит в MVP

```
✅ MUST HAVE для запуска:

1. Регистрация/вход
2. Создание узла (название, категория)
3. Список узлов
4. Отметка "выполнено сегодня"
5. Подсчёт стрика
6. Визуальный граф (базовый)
7. Дашборд со статистикой
8. Мобильная адаптивность
```

### 9.2 Что НЕ входит в MVP

```
❌ WON'T HAVE для первого запуска:

1. Email уведомления
2. Telegram бот
3. Мобильное приложение
4. Публичные профили
5. Социальные фичи
6. Экспорт/импорт
7. Темная тема (если только одна)
8. Мультиязычность
9. Платные подписки
10. Команды/шеринг
```

### B. Schema SQL для Supabase

```sql
-- Профили пользователей
CREATE TABLE profiles (
  id UUID REFERENCES auth.users PRIMARY KEY,
  email TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Узлы (привычки)
CREATE TABLE nodes (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT,
  category TEXT NOT NULL DEFAULT 'health',
  frequency TEXT NOT NULL DEFAULT 'daily',
  color TEXT,
  icon TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Импульсы (выполнения)
CREATE TABLE impulses (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  node_id UUID REFERENCES nodes(id) ON DELETE CASCADE,
  completed_at DATE NOT NULL DEFAULT CURRENT_DATE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  UNIQUE(node_id, completed_at)
);

-- Связи между узлами
CREATE TABLE connections (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  from_node_id UUID REFERENCES nodes(id) ON DELETE CASCADE,
  to_node_id UUID REFERENCES nodes(id) ON DELETE CASCADE,
  type TEXT NOT NULL DEFAULT 'enhances',
  strength INTEGER DEFAULT 1,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CHECK (from_node_id != to_node_id)
);

-- RLS Policies (безопасность)
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE nodes ENABLE ROW LEVEL SECURITY;
ALTER TABLE impulses ENABLE ROW LEVEL SECURITY;
ALTER TABLE connections ENABLE ROW LEVEL SECURITY;

-- Policies для profiles
CREATE POLICY "Users can view own profile"
  ON profiles FOR SELECT
  USING (auth.uid() = id);

-- Policies для nodes
CREATE POLICY "Users can view own nodes"
  ON nodes FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can insert own nodes"
  ON nodes FOR INSERT
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can update own nodes"
  ON nodes FOR UPDATE
  USING (auth.uid() = user_id);

CREATE POLICY "Users can delete own nodes"
  ON nodes FOR DELETE
  USING (auth.uid() = user_id);

-- Аналогично для impulses и connections...
```
